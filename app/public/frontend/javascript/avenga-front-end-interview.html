Технічна співбесіда Avenga

01-intro: Основи HTML / CSS / Browser
1. Як можна знайти посилання в тексті на конкретний ресурс за допомогою CSS?

Використовуються селектори атрибутів href.

a[href="https://example.com"] {}
a[href^="https"] {}
a[href$=".pdf"] {}
a[href*="google"] {}

Follow-up: у чому різниця між ^, $, *?

Відповідь: починається з / закінчується на / містить підрядок.

2. Як можна вирівняти елемент по горизонталі та вертикалі?

Flexbox — найпопулярніший спосіб.

.container {
  display: flex;
  justify-content: center;
  align-items: center;
}

Grid — зручно для двовимірних layout.

.container {
  display: grid;
  place-items: center;
}

Position + transform — класичний спосіб.

.box {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}

Margin auto — горизонтальне вирівнювання блоку.

.box {
  width: 200px;
  margin: 0 auto;
}

Text-align — для inline / inline-block.

.container {
  text-align: center;
}

Follow-up: який спосіб найкращий?

Відповідь: Flex або Grid — сучасні та універсальні.

3. Як відбувається завантаження сторінки в браузері?

1. Браузер робить HTTP-запит за HTML.

2. HTML парситься → створюється DOM.

3. CSS парситься → створюється CSSOM.

4. DOM + CSSOM → Render Tree.

5. Layout → Paint → Composite.

Follow-up: що блокує рендер?

Відповідь: CSS та синхронний JavaScript.

4. Як можна пришвидшити завантаження сторінки? Які є основні методи оптимізації?

Мета оптимізації — швидше показати користувачу перший екран і зменшити час завантаження ресурсів.

Мінімізація CSS / JS — зменшення розміру файлів шляхом видалення пробілів, коментарів і зайвого коду.

Code splitting — розбиття JS на чанки, щоб не завантажувати весь код одразу.

Lazy loading — відкладене завантаження зображень, компонентів або сторінок.

Кешування — повторне використання ресурсів через browser cache та HTTP-заголовки.

Оптимізація зображень — стиснення, правильні формати (WebP, AVIF), адаптивні розміри.

CDN — віддача файлів з найближчого до користувача сервера.

Follow-up: що оптимізуєш в першу чергу?

Відповідь: critical CSS, зображення, початковий JS-бандл.

5. Що таке Critical CSS?

Critical CSS — це стилі, необхідні для відображення першого екрану (above-the-fold).

Інлайниться в <head>, щоб браузер міг швидко відмалювати сторінку, а решта CSS підвантажується пізніше.

Follow-up: навіщо це потрібно?

Відповідь: для покращення FCP та швидшого рендеру.

6. Як працює lazy loading?

Lazy loading — це відкладене завантаження ресурсів, які не потрібні одразу.

Зображення, компоненти або JS-код підвантажуються лише тоді, коли потрапляють у viewport або потрібні користувачу. 

<img src="photo.jpg" loading="lazy" />

Follow-up: де не варто використовувати?

Відповідь: для критичного контенту першого екрану.

01-intro: Основи зображень
8. Які типи зображень ти знаєш? Які переваги та недоліки кожного з них?

JPEG / JPG — растровий формат з втратами.

➕ Малий розмір файлу, добре підходить для фото.

➖ Втрата якості при стисненні, немає прозорості.

PNG — растровий формат без втрат.

➕ Підтримує прозорість, висока якість.

➖ Більший розмір файлу порівняно з JPG.

SVG — векторний формат.

➕ Масштабується без втрати якості, малий розмір для іконок.

➖ Погано підходить для фотографій.

WebP — сучасний растровий формат.

➕ Менший розмір, ніж JPG/PNG, підтримує прозорість.

➖ Потрібна перевірка підтримки старими браузерами.

AVIF — сучасний растровий формат нового покоління.

➕ Найкраще стиснення і якість.

➖ Підтримка ще не всюди і повільніше кодується.

Follow-up: який формат обереш для фото на сайті?

Відповідь: WebP або AVIF з fallback на JPG.

9. Різниця між растровим та векторним зображенням?

Растрові зображення складаються з пікселів.

При збільшенні втрачають якість.

Приклади: JPG, PNG, WebP, AVIF.

Векторні зображення описуються формулами та кривими.

Масштабуються без втрати якості.

Приклади: SVG.

Follow-up: що краще для іконок?

Відповідь: SVG — легкий і масштабований.

06-images: Responsive images, Retina, Optimization
1. Як можна підвантажувати картинки під різні роздільні здатності екранів?

Основний спосіб — використання srcset та sizes, щоб браузер сам обрав оптимальне зображення. 

<img
  src="image-400.jpg"
  srcset="
    image-400.jpg 400w,
    image-800.jpg 800w,
    image-1600.jpg 1600w
  "
  sizes="(max-width: 600px) 100vw, 50vw"
  alt=""
/>

Браузер враховує:

• ширину екрана

• щільність пікселів

• значення sizes

Follow-up: хто вирішує, яку картинку завантажити?

Відповідь: браузер, а не розробник.

2. Як ще можна підвантажувати різні картинки? (окрім srcset)

Через тег <picture>, який дозволяє комбінувати формати та media-запити. 

<picture>
  <source srcset="image.webp" type="image/webp">
  <source srcset="image.jpg" type="image/jpeg">
  <img src="image.jpg" alt="">
</picture>

Також можна використовувати media queries для вибору зображення під конкретні умови. 

<picture>
  <source media="(max-width: 600px)" srcset="mobile.jpg">
  <source media="(min-width: 601px)" srcset="desktop.jpg">
  <img src="desktop.jpg" alt="">
</picture>

Follow-up: коли picture краще за srcset?

Відповідь: коли потрібні різні формати або різний контент для breakpoint’ів.

3. Яку роль відіграють формати зображень (WebP, JPG, PNG)?

WebP — сучасний формат з кращим стисненням, ніж JPG і PNG.

• менший розмір файлу

• підтримка прозорості

• підходить для фото і UI

Зазвичай використовують WebP з fallback на JPG або PNG через <picture>.

4. Що таке Retina та як працювати з екранами з високою щільністю пікселів?

Retina — це екрани з високою щільністю пікселів, де на один CSS-піксель припадає кілька фізичних пікселів.

Щоб зображення виглядали чітко, потрібно завантажувати картинки з більшою роздільною здатністю. 

<img
  src="icon.png"
  srcset="icon.png 1x, icon@2x.png 2x"
  alt=""
/>

Що таке DPI:

DPI (dots per inch) — кількість пікселів на дюйм. Чим вище DPI, тим більше деталей може відобразити екран.

На екранах з високим DPI стандартні картинки виглядають розмитими, якщо не використати 2x або 3x версії.

Follow-up: що краще для ретини — PNG чи SVG?

Відповідь: SVG, бо масштабується без втрати якості.

10. PNG — це який тип зображення?

PNG — це растровий формат зображення з безвтратним стисненням.

Підтримує прозорість (alpha channel) та використовується для іконок, UI-елементів, скріншотів.

Follow-up: чому PNG важчий за JPG?

Відповідь: тому що не використовує стиснення з втратами.

02-js-basics: Основи JavaScript
1. Як можна оголосити функції в JS?

Function Declaration:

function sum(a, b) {
  return a + b;
}

Function Expression:

const sum = function(a, b) {
  return a + b;
};

Arrow Function:

const sum = (a, b) => a + b;

Follow-up: який спосіб використовують найчастіше?

Відповідь: arrow function та function expression.

2. Різниця між function expression, arrow function та function declaration?

Function Declaration — піднімається через hoisting.

Function Expression — не піднімається.

Arrow Function — не має власного this, arguments.

Follow-up: чи можна використовувати arrow як constructor?

Відповідь: ні, у arrow немає this та prototype.

3. Що таке hoisting?

Hoisting — це механізм, за яким оголошення змінних і функцій піднімаються на початок області видимості. 

console.log(a); // undefined
var a = 5;

let та const hoisting мають, але доступні лише після ініціалізації (TDZ).

Follow-up: чи hoisting — це реальне переміщення коду?

Відповідь: ні, це поведінка execution context.

4. Що таке замикання (closure) в JS?

Closure — це доступ функції до змінних із зовнішньої області після завершення її виконання. 

function counter() {
  let count = 0;
  return function () {
    count++;
    return count;
  };
}

const inc = counter();
inc(); // 1
inc(); // 2

Follow-up: де використовується?

Відповідь: приватні змінні, callbacks, React hooks.

5. Що таке контекст виконання (execution context) в JS?

Execution context — це середовище, в якому виконується код.

Містить:

• змінні

• значення this

• scope chain

Типи: Global, Function, Eval.

Follow-up: скільки global context?

Відповідь: один на програму.

6. Що таке this в JS? Як визначається його значення?

this — це контекст виклику функції.

Метод обʼєкта: this → обʼєкт.

Звичайна функція: undefined у strict mode.

Arrow function: бере this із зовнішнього контексту.

call/apply/bind: задається вручну.

Follow-up: від чого залежить this?

Відповідь: від способу виклику, а не оголошення.

7. Що таке глобальний об'єкт в JS? Який він у браузері та Node.js?

Глобальний обʼєкт — це обʼєкт, який містить глобальні змінні та функції.

Browser: window

Node.js: global

Універсальний: globalThis

Follow-up: навіщо globalThis?

Відповідь: для кросплатформенного коду.

8. Що належить до глобальної області видимості (global scope) в window?

• глобальні змінні (var)

• глобальні функції

• document, location, history

• setTimeout, fetch

Follow-up: чи потрапляють let і const у window?

Відповідь: ні.

9. Як можна задати this для функції вручну?

Через call, apply, bind.

fn.call(obj, a, b);
fn.apply(obj, [a, b]);
const bound = fn.bind(obj);

Follow-up: який виконує одразу?

Відповідь: call та apply.

10. В чому різниця між bind та call / apply?

call / apply — викликають функцію одразу.

bind — повертає нову функцію з привʼязаним this.

Різниця call/apply: аргументи окремо / масивом.

11. Які способи роботи з асинхронним кодом в JS ти знаєш?

Callbacks — базовий, але може призводити до callback hell.

Promises

fetch(url)
  .then(res => res.json())
  .then(data => console.log(data));

async / await — синтаксичний цукор над Promise.

const data = await fetch(url);

Follow-up: як обробляти помилки?

Відповідь: .catch() або try/catch.

03-async-patterns: Observer, Events, MobX/Saga
1. Чи працював з MobX або Redux-Saga? Як це повʼязано з асинхронним кодом?

MobX — state management на базі реактивності: observable + observer. Асинхронність зазвичай роблять в actions (async/await), а UI автоматично реагує на зміни.

Redux-Saga — middleware для Redux, яке керує async потоками через generator-функції (effects типу takeLatest, call, put). Це відділяє async-логіку від компонентів і reducer’ів.

Плюси: краще структурування async, менше логіки в компонентах, контроль над race conditions.

Мінуси: складність, більше абстракцій, треба домовленості в команді.

Follow-up: що б обрав для простого проєкту?

Відповідь: RTK Query / async thunks / React Query; Saga — коли потрібні складні потоки.

2. Які патерни проєктування ти знаєш?

Creational: Factory, Singleton.

Structural: Decorator, Adapter.

Behavioral: Observer, Strategy.

Follow-up: який найчастіше в UI?

Відповідь: Observer (події/стан), Strategy (різні алгоритми), Decorator (HOC/middleware).

3. Як працює Observer pattern?

Observer pattern — це взаємодія «один-до-багатьох», де один обʼєкт (Subject) повідомляє підписників (Observers) про події або зміни.

У браузері класичний приклад Observer — це addEventListener, який реалізований через callbacks.

Ролі:

• Subject — DOM-елемент (кнопка)

• Observer — callback-функція

• Подія — click, input, scroll

Callback у цьому випадку і є Observer — функція, яка чекає на подію і реагує на неї.

Як це працює крок за кроком:

1. Observer підписується на подію через addEventListener

2. Браузер зберігає callback у внутрішньому списку слухачів

3. При виникненні події браузер проходить по списку listeners

4. Кожен callback викликається з обʼєктом події

const onClick = () => {
  console.log('button clicked');
};

button.addEventListener('click', onClick);

Коли користувач клікає кнопку, браузер автоматично повідомляє всіх підписників — викликає їх callbacks.

Відписка (unsubscribe):

button.removeEventListener('click', onClick);

Відписка є важливою частиною Observer pattern, інакше callbacks продовжать викликатися та можуть призводити до витоків памʼяті.

Чому це Observer pattern:

• Subject не знає, хто саме на нього підписаний
• може бути будь-яка кількість observers
• реакція відбувається лише при події
• звʼязок між обʼєктами слабкий (loose coupling)

Практичні приклади застосування:

• DOM events (addEventListener)

• React (зміна state → rerender компонентів)

• MobX (observable → observer)

• RxJS (subscribe / next)

Follow-up: коли викликається callback?

Відповідь: коли подія відбувається і доходить до елемента на відповідній фазі (capturing / target / bubbling).

4. Якщо підписався на подію, як відписатися від неї?

У DOM — через removeEventListener з тим самим типом події і тим самим callback (та опціями, якщо використовувались).

const onClick = () => console.log('click');

btn.addEventListener('click', onClick);
btn.removeEventListener('click', onClick);

Важливо: якщо передати нову функцію (інший reference) — відписка не спрацює.

5. Якщо підписався на подію — що відбувається далі?

Браузер додає callback у список слухачів (listeners) для конкретного елемента і типу події.

Коли подія трапляється, браузер проходить по списку listeners і викликає їх у порядку реєстрації.

Подія проходить фази: capturing → target → bubbling (якщо bubbling є).

Follow-up: що таке event delegation?

Відповідь: ставимо один listener на батька і ловимо події від дітей через bubbling.

6. Дати розгорнуту відповідь: "Як працює Observer?"

Observer — це шаблон взаємодії "один-до-багатьох": один Subject повідомляє багато Observer’ів.

Навіщо: розв’язує жорстку залежність — Subject не знає деталей підписників, тільки викликає їх.

Структура:

• Subject зберігає список підписників

• методи: subscribe, unsubscribe, notify

• Observer — це callback/об’єкт з методом update

Приклад з життям: кнопка (Subject) → обробники кліку (Observers); store → компоненти.

Типова помилка: не відписуватися → витоки памʼяті (особливо в SPA).

7. Розповісти про addEventListener та принцип роботи цього патерну.

addEventListener(type, callback, options) реєструє callback як слухача події.

type: 'click', 'input', 'scroll', ...

callback: функція, яка отримає об’єкт події event.

options: налаштування фази і поведінки.

btn.addEventListener('click', (e) => {
  console.log(e.type);        // 'click'
  console.log(e.target);      // елемент, по якому клікнули
});

Фази події:

• Capturing: зверху вниз (якщо { capture: true })

• Target: на цільовому елементі

• Bubbling: знизу вверх (за замовчуванням)

Корисні options:

• { once: true } — виконається 1 раз

• { passive: true } — не викликаємо preventDefault (краще для scroll)

• { capture: true } — слухати на фазі capturing

window.addEventListener('scroll', onScroll, { passive: true });
btn.addEventListener('click', onClick, { once: true });
8. Практичне застосування Observer: де реально використовується?

DOM events: click/input/scroll — класичний Observer.

State management: MobX (observable → observer), React state (компонент реагує на зміни).

Streams: RxJS (subscribe → next/error/complete).

Pub/Sub в додатку: event bus для внутрішніх подій.

// RxJS ідея (спрощено)
const observers = [];
const subscribe = (fn) => observers.push(fn);
const next = (v) => observers.forEach(fn => fn(v));

Follow-up: чому це корисно?

Відповідь: слабке зв’язування модулів і реакція на зміни без прямого виклику один одного.

9. Коли викличиться callback, переданий в addEventListener?

Коли відбудеться відповідна подія (наприклад, 'click') і подія дійде до елемента на потрібній фазі (capturing/target/bubbling).

Callback викликається з обʼєктом події event і з this, що вказує на елемент (для звичайної функції).

btn.addEventListener('click', function (e) {
  console.log(this === btn); // true
});

btn.addEventListener('click', (e) => {
  // arrow бере this ззовні
});

Follow-up: як зупинити спливання?

Відповідь: e.stopPropagation() (а дефолтну дію — e.preventDefault()).

04-react-core: Ререндер, Virtual DOM, Controlled / Uncontrolled
1. Що зробити, щоб компонент ререндерився в React?

Компонент у React ререндериться, коли змінюються його вхідні дані або контекст.

Основні причини ререндеру:

• при зміні props

• при оновленні state через useState (setState)

• при зміні значення Context

• при ререндері батьківського компонента

• при зміні значення key між рендерами — React перемонтовує компонент

Follow-up: чи ререндер завжди означає оновлення DOM?

Відповідь: ні, React може вирішити, що DOM не потребує змін.

2. Як React знає, що саме потрібно ререндерити?

React порівнює попередній і новий результат рендеру за допомогою Virtual DOM.

Він визначає мінімальний набір змін, які потрібно застосувати до реального DOM.

Ключова ідея: не оновлювати весь DOM, а лише те, що змінилось.

3. Що таке Virtual DOM?

Virtual DOM — це легковажне JS-представлення реального DOM.

Кожен рендер створює нове дерево Virtual DOM, яке потім порівнюється з попереднім.

Навіщо:

• реальний DOM повільний

• JS-операції значно швидші

• можна виконати batch-оновлення

4. Як Virtual DOM працює під капотом? Як відбувається порівняння?

Після кожного ререндеру React:

1. Створює нове дерево Virtual DOM

2. Порівнює його з попереднім (diffing)

3. Обчислює різницю

4. Застосовує мінімальні зміни до реального DOM

Основні евристики diffing алгоритму:

• елементи різного типу → повна заміна піддерева

• однакові типи → оновлюються лише props

• списки порівнюються за key

Приклад:

// було
<li key="1">A</li>
<li key="2">B</li>

// стало
<li key="2">B</li>
<li key="1">A</li>

Завдяки key React не перевмонтуює елементи, а лише міняє їх місцями.

Важливо: без key React може зробити зайві DOM-операції.

5. Що таке reconciliation в React?

Reconciliation — це процес узгодження старого і нового Virtual DOM.

Саме під час reconciliation React вирішує, що потрібно оновити, змінити або видалити.

Простими словами: reconciliation = diffing + update.

6. Різниця між контрольованим та неконтрольованим компонентом?

Контрольований компонент — стан інпуту зберігається в React.

<input value={value} onChange={e => setValue(e.target.value)} />

Неконтрольований компонент — стан зберігається в DOM.

<input defaultValue="test" ref={inputRef} />

Головна різниця: хто керує станом — React чи браузер.

7. В чому краще використовувати контрольовані компоненти?

Контрольовані компоненти краще використовувати, коли потрібно:

• валідація в реальному часі

• синхронізація значень

• умовна логіка UI

• повний контроль над формою

Типові кейси: auth, фільтри, step-форми.

8. Коли доцільно використовувати неконтрольовані компоненти?

У неконтрольованих компонентах React створює DOM-елементи, а браузер сам керує їх станом.

Корисно для:

• великих форм з багатьма інпутами, де не потрібно контролювати кожен символ

• інтеграції зі сторонніми бібліотеками (date picker, rich text editor)

• простих одноразових форм

Плюс: менше ререндерів.

Мінус: менше контролю та складніша логіка.

05-react-hooks: Hooks, Context, Optimization
1. Якими хуками ти користуєшся найчастіше?

Найчастіше використовую базові хуки: useState, useEffect, useRef, useMemo, useCallback, useContext.

Коротко по призначенню:

• useState — локальний стан

• useEffect — побічні ефекти

• useRef — DOM або мутабельне значення без ререндеру

• useMemo — мемоізація значень

• useCallback — мемоізація функцій

• useContext — доступ до глобальних даних

2. Для яких цілей використовується useMemo?

useMemo використовується для мемоізації результату обчислень між рендерами.

Навіщо:

• уникнути повторних важких обчислень

• зменшити навантаження при частих ререндерах

• оптимізувати продуктивність

const total = useMemo(() => {
  return items.reduce((sum, i) => sum + i.price, 0);
}, [items]);

Важливо:

useMemo працює зі значеннями, а не з функціями (для цього є useCallback).

Follow-up: чи варто обгортати все в useMemo?

Відповідь: ні, useMemo має оверхед і потрібен лише там, де є реальна проблема з продуктивністю.

3. Для чого використовується useContext?

useContext дозволяє отримувати дані з React Context без прокидування props через багато рівнів.

Типові кейси:

• тема (theme)

• локалізація

• auth / user

• feature flags

const ThemeContext = createContext();

const theme = useContext(ThemeContext);
4. Де зберігаються дані useContext в React?

Дані зберігаються у value провайдера (<Context.Provider>). 

<AuthContext.Provider value={user}>
  <App />
</AuthContext.Provider>

Усі дочірні компоненти отримують доступ до цього значення через useContext.

5. Context — це global state?

Так, Context можна вважати глобальним станом, але він не є повноцінним state-manager’ом.

Відмінності:

• немає middleware

• немає інструментів дебагу

• при зміні value ререндеряться всі споживачі

6. Плюси та мінуси useContext

Плюси:

• простий API

• немає prop drilling

• вбудований у React

Мінуси:

• всі споживачі ререндеряться при зміні value

• не підходить для часто змінюваного стану

Висновок: useContext варто використовувати для даних, які змінюються рідко.

7. Які є способи оптимізації React-додатку?

• React.memo

• useMemo, useCallback

• правильні key у списках

• code splitting / lazy loading

• уникання зайвого state

• розділення логіки і UI

Follow-up: з чого починати?

Відповідь: з аналізу ререндерів (React DevTools).

8. Що таке кастомні хуки в React?

Кастомний хук — це функція, яка використовує інші хуки і інкапсулює логіку.

Правила:

• імʼя починається з use

• можна викликати інші хуки

9. Чи писав ти кастомні хуки? Наведи приклад. 
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = () => setValue(v => !v);
  return [value, toggle];
}

Такий хук дозволяє перевикористовувати логіку без дублювання коду.

10. Чим кастомний хук відрізняється від звичайної функції?

Кастомний хук:

• може використовувати React hooks

• підпорядковується правилам хуків

Звичайна функція:

• не має доступу до хуків

• не зберігає React-стан

Ключова різниця: хуки інтегровані в lifecycle React.

11. Розкажи більше про useEffect

useEffect використовується для побічних ефектів після рендеру компонента.

Типові кейси:

• запити до API

• підписки

• таймери

• синхронізація з DOM

Залежності:

• [] — лише при монтуванні

• [dep] — при зміні залежності

• без масиву — після кожного рендеру

useEffect(() => {
  fetchData();
  return () => cleanup();
}, [id]);

Важливо: неправильні залежності → infinite loop.

07-browser-storage: Детальніше про Browser Storage
1. Які стореджі є в браузері? Детальна різниця між ними

Основні browser storage:

localStorage

• Розмір: ~5–10 MB на домен

• Час життя: безстроково (поки не видалять вручну)

• Доступ: всі вкладки одного домену

• Тип даних: лише рядки

• Не надсилається на сервер

sessionStorage

• Розмір: ~5 MB

• Час життя: поки відкрита вкладка

• Доступ: тільки в межах однієї вкладки

• Тип даних: лише рядки

• Не надсилається на сервер

Cookies

• Розмір: ~4 KB на cookie

• Кількість: ~20–50 cookies на домен

• Час життя: задається (session або persistent)

• Автоматично надсилаються на сервер

2. Детально: localStorage

localStorage — синхронне key-value сховище, доступне через JavaScript. 

localStorage.setItem('theme', 'dark');
localStorage.getItem('theme');

Особливості:

• блокує main thread (синхронний API)

• не підходить для великих або чутливих даних

• дані переживають перезавантаження сторінки

Типові кейси:

• theme

• language

• UI preferences

3. Детально: sessionStorage

sessionStorage працює так само, як localStorage, але обмежений життєвим циклом вкладки.

Особливість:

навіть якщо відкрити ту саму сторінку в новій вкладці — дані не будуть спільними.

Типові кейси:

• кроки multi-step форми

• тимчасові фільтри

• draft-стан

4. Детально: cookies

Cookies — дані, які браузер зберігає і може надсилати на сервер разом з HTTP-запитами.

Час життя:

• session cookies — до закриття браузера

• persistent cookies — за expires або max-age

Обмеження:

• маленький розмір (~4KB)

• кожен HTTP-запит стає важчим

Безпека:

• HttpOnly — недоступна з JS (захист від XSS)

• Secure — лише HTTPS

• SameSite — захист від CSRF

Set-Cookie: sessionId=abc;
  HttpOnly;
  Secure;
  SameSite=Strict;
  Max-Age=3600;

Follow-up: де безпечніше зберігати auth-токен?

Відповідь: HttpOnly cookie — безпечніше, ніж localStorage.

5. Коротке порівняння (як на співбесіді)

localStorage — великий, довгоживучий, тільки клієнт

sessionStorage — тимчасовий, на вкладку

cookies — маленькі, але доступні серверу
