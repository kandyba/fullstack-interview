Що таке SOLID?

SOLID - це п'ять принципів об'єктно-орієнтованого дизайну, які допомагають створювати більш підтримуваний, гнучкий та масштабований код.

  • S - Single Responsibility Principle (Принцип єдиної відповідальності)
  • O - Open/Closed Principle (Принцип відкритості/закритості)
  • L - Liskov Substitution Principle (Принцип підстановки Барбари Лісков)
  • I - Interface Segregation Principle (Принцип розділення інтерфейсів)
  • D - Dependency Inversion Principle (Принцип інверсії залежностей)
Single Responsibility Principle (SRP) - Принцип єдиної відповідальності

Визначення: Кожен клас повинен мати лише одну причину для зміни, тобто одну відповідальність.

❌ Погано: Клас робить забагато

class UserService {
  saveUser(user) { /* зберігає в БД */ }
  sendEmail(user) { /* відправляє email */ }
  generateReport(user) { /* генерує звіт */ }
  validateUser(user) { /* валідація */ }
}

✅ Добре: Кожен клас має одну відповідальність

class UserRepository {
  save(user) { /* тільки робота з БД */ }
}

class EmailService {
  send(user) { /* тільки відправка email */ }
}

class ReportGenerator {
  generate(user) { /* тільки генерація звітів */ }
}

class UserValidator {
  validate(user) { /* тільки валідація */ }
}
Open/Closed Principle (OCP) - Принцип відкритості/закритості

Визначення: Класи повинні бути відкриті для розширення, але закриті для модифікації.

❌ Погано: Потрібно змінювати існуючий код

class PaymentProcessor {
  processPayment(type, amount) {
    if (type === 'credit') {
      // логіка кредитки
    } else if (type === 'paypal') {
      // логіка PayPal
    } else if (type === 'crypto') {
      // додаємо новий тип - змінюємо клас!
    }
  }
}

✅ Добре: Розширюємо через нові класи

class PaymentMethod {
  process(amount) {
    throw new Error('Must be implemented');
  }
}

class CreditCardPayment extends PaymentMethod {
  process(amount) { /* логіка кредитки */ }
}

class PayPalPayment extends PaymentMethod {
  process(amount) { /* логіка PayPal */ }
}

class CryptoPayment extends PaymentMethod {
  process(amount) { /* нова функціональність без зміни існуючих класів */ }
}
Liskov Substitution Principle (LSP) - Принцип підстановки Барбари Лісков

Визначення: Об'єкти підкласів повинні замінювати об'єкти базових класів без порушення роботи програми.

❌ Погано: Підклас порушує контракт батьківського класу

class Bird {
  fly() { return 'Flying'; }
}

class Penguin extends Bird {
  fly() {
    throw new Error('Penguins cannot fly!'); // порушення LSP
  }
}

✅ Добре: Правильна ієрархія класів

class Bird {
  layEggs() { return 'Laying eggs'; }
}

class FlyingBird extends Bird {
  fly() { return 'Flying'; }
}

class Penguin extends Bird {
  swim() { return 'Swimming'; }
}

class Eagle extends FlyingBird {
  // може літати і нести яйця
}
Interface Segregation Principle (ISP) - Принцип розділення інтерфейсів

Визначення: Клієнти не повинні залежати від інтерфейсів, які вони не використовують.

❌ Погано: Великий інтерфейс з непотрібними методами

class Machine {
  print() { }
  scan() { }
  fax() { }
  staple() { }
}

class SimplePrinter extends Machine {
  print() { /* працює */ }
  scan() { throw new Error('Not supported'); }
  fax() { throw new Error('Not supported'); }
  staple() { throw new Error('Not supported'); }
}

✅ Добре: Маленькі специфічні інтерфейси

class Printer {
  print() { }
}

class Scanner {
  scan() { }
}

class FaxMachine {
  fax() { }
}

// SimplePrinter implements only Printer
class SimplePrinter extends Printer {
  print() { /* працює */ }
}

// Multifunctional implements all needed interfaces
class MultifunctionalMachine extends Printer {
  constructor() {
    super();
    this.scanner = new Scanner();
    this.fax = new FaxMachine();
  }
}
Dependency Inversion Principle (DIP) - Принцип інверсії залежностей

Визначення: Модулі верхнього рівня не повинні залежати від модулів нижнього рівня. Обидва повинні залежати від абстракцій.

❌ Погано: Жорстка залежність від конкретної реалізації

class MySQLDatabase {
  save(data) { /* збереження в MySQL */ }
}

class UserRepository {
  constructor() {
    this.db = new MySQLDatabase(); // жорстка залежність
  }
  
  saveUser(user) {
    this.db.save(user);
  }
}

✅ Добре: Залежність від абстракції (інтерфейсу)

// Абстракція
class Database {
  save(data) {
    throw new Error('Must be implemented');
  }
}

// Конкретні реалізації
class MySQLDatabase extends Database {
  save(data) { /* збереження в MySQL */ }
}

class MongoDatabase extends Database {
  save(data) { /* збереження в MongoDB */ }
}

// Залежність від абстракції через Dependency Injection
class UserRepository {
  constructor(database) {
    this.db = database; // приймає будь-яку Database
  }
  
  saveUser(user) {
    this.db.save(user);
  }
}

// Використання
const mysqlRepo = new UserRepository(new MySQLDatabase());
const mongoRepo = new UserRepository(new MongoDatabase());
Переваги використання SOLID принципів
  • Підтримуваність - легше розуміти та змінювати код
  • Тестованість - простіше писати unit тести
  • Масштабованість - легко додавати нову функціональність
  • Гнучкість - можна легко змінювати реалізації
  • Повторне використання - компоненти більш незалежні
  • Менше помилок - зміни в одному місці не ламають інше
Як застосовувати SOLID в JavaScript/TypeScript?

SRP: Розділяйте компоненти на менші з одною відповідальністю

OCP: Використовуйте composition, hooks, HOC для розширення

LSP: Дотримуйтесь очікуваної поведінки при наслідуванні

ISP: Створюйте маленькі специфічні props інтерфейси

DIP: Використовуйте Dependency Injection (React Context, DI frameworks)
