Запитання та відповіді з CI/CD

Що таке Continuous Integration (CI)
    CI = Постійне інтегрування коду в спільну гілку (main/master).

    Суть CI
    Кожен пуш → автоматично запускається build і тести.
    Мета: піймати помилки раніше, не ламати загальну гілку.

    Навіщо це?
    Менше конфліктів при злитті.
    Код завжди у стабільному стані.
    Команда рухається швидше, бо кожен інтегрує код регулярно.

Типовий CI-процес
Dev працює локально.
Пушить у репозиторій.
CI запускає збірку.
Запускаються тести.
Якщо все ок → зміна інтегрується.
Може автоматично йти на UAT/test-сервер.
Continuous Delivery (CD — delivery)
  CD Delivery = автоматичне "доставлення" нової версії до середовища UAT.

  Тобто:
  CI зробив build + тести
  CD Delivery автоматично деплоїть на UAT
  А потім деплой на прод — вручну (кнопка, рішення менеджера)

  Для чого?
  QA може перевірити фічу в реальному середовищі.
  Продакшн не ламається відразу.
  Бізнес контролює реліз.
Continuous Deployment (CD — deployment)
Continuous Deployment = повна автоматизація деплою:
Це найвищий рівень автоматизації.

Відмінність:

Підхід	               Хто запускає прод деплой
Continuous Delivery	   людина (manual trigger)
Continuous Deployment	 машина (автоматично)
CI/CD Pipeline (що таке pipeline)
Pipeline = ланцюжок автоматизованих кроків:
commit → build → test → deploy (UAT → prod)
  Типовий pipeline:
  Push to repo
  Build code
  Run tests
  Deploy to UAT
  Approve release
  Deploy to PROD
Що таке GitHub Actions?
Сервер автоматизації всередині GitHub. Дає змогу запускати скрипти при пуші, PR, релізі.
Що таке GitHub Actions — простими словами
GitHub Actions складаються з:

Поняття	            Значення
Workflow	          файл автоматизації .github/workflows/...yml
Job	                набір кроків, запускається на віртуалці
Step	              одна дія: команда або готовий action
Actions	            готові шматки логіки (чуже “uses: …”)
Runner	            машина (Ubuntu / Windows / Mac), де крутиться код
Event	              подія (push, PR, manual, cron), яка запускає workflow
Типові застосування GitHub Actions
CI:
build JS проекту
запуск Jest / Vitest
ESLint
Prettier

CD:
деплой на:
Vercel
Netlify
AWS / GCP / Azure
Docker Hub
1. Розробники додають нові зміни.
2. Зміни тестуються через автоматизовані тести.
3. Після успішного тестування код інтегрується в основну гілку.
4. За допомогою CD зміни можуть бути автоматично або вручну доставлені до продуктивного середовища.
Чому це важливо?
Прискорює розробку
Мінімізує ризики
Гарантує стабільність
Автоматизація економить години роботи
Дає можливість робити релізи кожного дня
Підтримує якісний та перевірений код
Шпаргалка, яку можна вивчити для співбесіди
CI: Автоматизація тестів і білду при кожному пуші — щоб інтеграція була безпечною. Continuous Delivery: Автоматичний деплой на UAT, а прод — вручну. Continuous Deployment: Автоматичний деплой на прод після UAT без участі людей. GitHub Actions: Це система автоматизації у GitHub, яка дозволяє запускати workflow-и при подіях (push, PR), використовуючи jobs, steps, runners та готові actions.
Що таке workflow / job / step / runner?
  Workflow — весь сценарій
  Job — група кроків
  Step — один крок
  Runner — віртуальна машина
