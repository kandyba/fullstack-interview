Запитання та відповіді для технічної співбесіди — Advanced Frontend (1–10)

1. Що таке Critical Rendering Path і як ти його оптимізуєш у SPA / SSR?

CRP — шлях від отримання HTML до першого пікселя: DOM → CSSOM → Render Tree → Layout → Paint.

Оптимізація: inline critical CSS, defer/async для JS, мінімізувати блокуючі ресурси, font-display: swap, компресія/HTTP2, зображення сучасних форматів, у SPA — code splitting; у SSR/SSG — швидший перший рендер.

<link rel="preload" as="style" href="/critical.css">
<link rel="stylesheet" href="/rest.css" media="print" onload="this.media='all'">
<script src="/main.js" defer></script>
2. Коли має сенс використовувати Web Workers і як вони впливають на main thread?

Коли: важкі обчислення (парсинг великих JSON, сортування, крипто, зображення/canvas) — щоб не блокувати UI.

Вплив: робота у фоновому потоці, main thread вільний, взаємодія через postMessage; доступу до DOM немає.

// main.js
const w = new Worker('worker.js');
w.postMessage(largeData);
w.onmessage = (e) => render(e.data);

// worker.js
self.onmessage = (e) => {
  const result = heavyCompute(e.data);
  self.postMessage(result);
};
3. Різниця між render blocking, layout thrashing і reflow — як це виявити та зменшити?

Render blocking: CSS/JS затримують побудову/рендер.

Layout thrashing: чергування читань/записів layout властивостей у циклах → множинні reflow.

Reflow: перерахунок геометрії, що може спричинити repaint.

Зменшення: групувати DOM-зміни, читати перед записами, використовувати transform/opacity, requestAnimationFrame, CSS contain.

// Погано (thrashing):
for (...) {
  const h = el.offsetHeight; // read
  el.style.height = (h + 10) + 'px'; // write
}
// Добре:
const h = el.offsetHeight;
requestAnimationFrame(() => { el.style.height = (h + 10) + 'px'; });
4. Що таке TTI (Time to Interactive) і як його виміряти (Lighthouse, Web Vitals, Performance API)?

TTI — момент, коли сторінка стає повністю інтерактивною: основні ресурси завантажені, обробка JS завершена, input відгук < 50мс.

Вимір: Lighthouse/Web Vitals. Поліпшення: code splitting, пріоритезація ресурсів, скорочення blocking JS, lazy для не-критичних частин.

// інструменти: Lighthouse (TTI), Web Vitals (FID/INP як наближені до інтерактивності)
5. Які стратегії кешування на клієнті ти знаєш (Cache First, SWR, stale-while-revalidate, network-only)?

Cache First (миттєво з кешу, оновлення потім), Network First (свіжі дані, fallback кеш), Stale-While-Revalidate (показує кеш + фонове оновлення), Network Only / Cache Only.

// Service Worker (Workbox):
workbox.routing.registerRoute(
  /\/api\/data/,
  new workbox.strategies.StaleWhileRevalidate()
);
6. Як працює preload, prefetch, dns-prefetch, preconnect — і коли що варто застосовувати?

preload — критичні ресурси зараз; prefetch — ймовірні майбутні; dns-prefetch — підготовка DNS; preconnect — ранній TCP/TLS handshake.

<link rel="preconnect" href="https://cdn.example.com" crossorigin>
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="preload" as="font" href="/fonts/main.woff2" crossorigin>
<link rel="prefetch" href="/next-chunk.js" as="script">
7. Як ти підходиш до code splitting і lazy loading? Коли краще розділяти, а коли об’єднувати?

Розділяти: великі маршрути/рідко використовувані фічі, адмінки, editors.

Об’єднувати: часто спільні та критичні частини (shared/vendor), щоб уникати надмірних запитів.

// React:
const Settings = React.lazy(() => import('./pages/Settings'));
<Suspense fallback={<Spinner />}><Settings /></Suspense>
8. Як оптимізувати перформанс React-додатка при великій кількості компонентів?

Мемоізація (React.memo, useMemo, useCallback), розбивати контексти, селектори стану, віртуалізація списків (React Window/Virtuoso), уникати зайвих re-render через стабільні пропи/keys, профілювання DevTools.

const Item = React.memo(function Item({ value }) { /* ... */ });
const list = useMemo(() => heavyTransform(data), [data]);

Tree shaking (sideEffects:false), import() для не-критичних модулів, заміна важких залежностей (lodash → lodash-es / нативне), splitChunks, gzip/brotli, аналіз пакетів.

// package.json
"sideEffects": false
// Аналіз:
npx webpack-bundle-analyzer dist/stats.json
10. Як працює reconciliation у React і коли він може стати вузьким місцем?

Reconciliation — diff нового й попереднього Virtual DOM і оновлення реального DOM з мінімальними змінами.

Вузькі місця: глибокі дерева, нестабільні keys, важкі рендер-функції. Рішення: мемоізація, стабільні ключі, розбиття великих компонентів, PureComponent/React.memo.

<List>{items.map(item => <Row key={item.id} data={item}/>)}</List>
11. Як працює Event Loop разом із microtasks і macrotasks у браузері?

Event Loop — механізм, що керує чергою виконання коду.

Macrotasks: setTimeout, setInterval, I/O, script, UI rendering.

Microtasks: Promise callbacks, MutationObserver, queueMicrotask.

Після кожної макрозадачі виконується вся черга microtasks. Потім — оновлення DOM.

console.log('A');
setTimeout(() => console.log('B'));
Promise.resolve().then(() => console.log('C'));
console.log('D');
// A D C B
12. Що таке task queue і rendering pipeline?

Task Queue — черга задач, які виконуються Event Loop’ом. Кожен task — один цикл обробки подій (JS, input, render).

Rendering pipeline: JavaScript → Style → Layout → Paint → Composite.

Якщо JS блокує main thread — фрейми пропускаються (jank). Використовують requestAnimationFrame для синхронізації з рендером.

13. Як працює garbage collection у JS і коли можуть з’являтися memory leaks?

JS використовує mark-and-sweep алгоритм: усі об’єкти, до яких є посилання, залишаються; інші — видаляються.

Memory leaks: незакриті таймери, замикання, глобальні посилання, неочищені listeners або refs у React.

useEffect(() => {
  const id = setInterval(() => doSomething(), 1000);
  return () => clearInterval(id); // важливо очистити
}, []);
14. Як браузер обробляє repaint і reflow, і як DevTools допомагає це відслідковувати?

Reflow: перерахунок геометрії елементів.

Repaint: оновлення вигляду (колір, фон) без зміни розміру.

У DevTools: вкладка Performance → Paint/Render/Composite events. Також у Rendering → “Paint flashing”.

15. Як працює Service Worker, lifecycle, та як його оновлювати без “білого екрана”?

Service Worker — проксі між додатком і мережею. Має lifecycle: install → activate → fetch.

Для оновлення без “білого екрану”: використовуй skipWaiting() та clients.claim(), а також стратегію “update in background”.

self.addEventListener('install', e => self.skipWaiting());
self.addEventListener('activate', e => self.clients.claim());
16. Що відбувається, коли ти вводиш URL у браузер і тиснеш Enter?

1. DNS lookup → IP.

2. TCP handshake (3-way) + TLS handshake (HTTPS).

3. Надсилається HTTP-запит.

4. Сервер відповідає → браузер завантажує HTML → будує DOM, CSSOM, JS.

5. Рендеринг: Layout → Paint → Composite → Interaction ready.

17. Як браузер вирішує, що можна паралельно завантажити, а що блокує рендеринг?

HTML парситься послідовно. CSS — render-blocking, JS — теж, якщо без defer/async.

Браузер має ліміт одночасних з’єднань (~6 на домен). preload і HTTP/2 дозволяють краще розпаралелити завантаження.

18. Що таке Render Tree, DOM Tree, CSSOM, і як вони пов’язані?

DOM Tree: структура HTML-елементів.

CSSOM: дерево стилів після парсингу CSS.

Render Tree: поєднання DOM + CSSOM (видимі елементи) → Layout → Paint.

HTML + CSS → DOM + CSSOM → Render Tree → Layout → Paint → Composite
19. Як відбувається compositing та чому GPU-рендеринг може бути повільним?

Compositing: об’єднання шарів після Paint. CSS властивості типу transform або opacity створюють GPU-слої.

Повільність: надмірна кількість шарів або великий обсяг текстур → переповнення відеопам’яті, контекстні перемикання GPU.

20. Як працює debounce/throttle і чому важливо їх застосовувати при scroll/resize подіях?

Debounce: викликає функцію після паузи. Throttle: обмежує виклики певним інтервалом.

Зменшує навантаження на main thread, запобігає “лагам” при подіях, що тригеряться часто.

// debounce
function debounce(fn, delay) {
  let t;
  return (...args) => {
    clearTimeout(t);
    t = setTimeout(() => fn(...args), delay);
  };
}

// throttle
function throttle(fn, limit) {
  let last = 0;
  return (...args) => {
    const now = Date.now();
    if (now - last >= limit) {
      last = now;
      fn(...args);
    }
  };
}
21. Як React 19 змінив роботу з асинхронними даними (use, Suspense, серверні компоненти)?

У React 19 з’явився use() — універсальний хук, який дозволяє очікувати Promise прямо у компоненті.

Suspense тепер підтримує асинхронні дані без додаткових бібліотек. Серверні компоненти можуть виконувати async-логіку до відправки HTML клієнту.

async function getUser() { return fetch('/api/user').then(r => r.json()); }

function Profile() {
  const user = use(getUser()); // автоматично чекає
  return <h2>{user.name}</h2>;
}
22. Як працюють Server Components і чому це важливо для продуктивності?

Server Components — компоненти, що рендеряться на сервері без відправки JS-бандлу клієнту.

Це зменшує розмір JS, прискорює TTI та дозволяє безпечно отримувати дані (доступ до DB, API) без витоку токенів.

React комбінує серверні й клієнтські компоненти у єдине дерево через streaming SSR.

23. Чим відрізняються hydration і streaming SSR?

Hydration — React “підключає” події до вже готового HTML з сервера.

Streaming SSR — сервер надсилає HTML потоково частинами, дозволяючи швидше відобразити UI.

Hydration завершується після завантаження JS-бандлу, тоді як streaming SSR дозволяє показати skeleton або layout раніше.

24. Як ти налагоджуєш memory leaks у React (наприклад, у useEffect)?

Використовую профілювання Performance / Memory DevTools або Sentry Performance.

У React — очищення у useEffect, перевірка неконтрольованих setState після unmount, закриття WebSocket і таймерів.

useEffect(() => {
  const ws = new WebSocket('wss://...');
  return () => ws.close(); // очищення
}, []);
25. Які ти використовуєш підходи до оптимізації ререндерів (React.memo, useCallback, useMemo)?

React.memo — мемоізація компонентів за props.

useMemo — кешує обчислення.

useCallback — стабілізує функції між рендерами.

Також — селектори стану (Reselect, Zustand), спліт контекстів, key-оптимізація.

const filtered = useMemo(() => data.filter(x => x.active), [data]);
const handleClick = useCallback(() => setOpen(true), []);
26. Як правильно організувати глобальний state management (Redux, Zustand, Jotai, Signals)?

Залежить від масштабу:

  • Redux — централізований state + middleware (redux-saga, thunk).
  • Zustand — легкий, без boilerplate, підтримує selectors.
  • Jotai / Signals — fine-grained реактивність, мінімальні ререндери.

Великі застосунки: Redux Toolkit; дрібні — Zustand або Signals.

27. Як ти тестуєш компоненти — unit vs integration?

Unit-тести: ізольовано, без DOM (Jest + Testing Library).

Integration-тести: реальна поведінка користувача (рендер, події, routing).

Підхід: Testing Library — фокус на поведінці, а не на реалізації.

test('shows greeting', () => {
  render(<App />);
  expect(screen.getByText(/hello/i)).toBeInTheDocument();
});
28. Як працює Error Boundary, і коли він реально рятує?

Error Boundary — класовий компонент, що ловить помилки у піддереві React і відображає fallback-UI, не ламаючи весь додаток.

Не ловить помилки у event-handlers або async коді.

class Boundary extends React.Component {
  state = { hasError: false };
  static getDerivedStateFromError() { return { hasError: true }; }
  render() {
    return this.state.hasError ? <ErrorScreen /> : this.props.children;
  }
}
29. Як реалізувати власні hooks для повторного використання логіки?

Створюю функції, що інкапсулюють повторну логіку — роботу з API, form state, scroll events, animation state тощо.

Назви завжди починаються з use, щоб React міг відстежувати їхній стан.

function useFetch(url) {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch(url).then(r => r.json()).then(setData);
  }, [url]);
  return data;
}
30. Як би ти спроєктував Design System або UI-бібліотеку під власний продукт?

Створив би atomic дизайн систему (Atoms → Molecules → Organisms → Templates).

Компоненти у вигляді Storybook, ізольовані стилі (Tailwind, CSS Modules, Styled Components).

Теми через CSS variables або context. Документація — через Storybook Docs.

// Button.tsx
export const Button = ({ variant = 'primary', ...props }) => (
  <button className={`btn btn--${variant}`} {...props} />

31. Як захистити SPA від XSS, CSRF, Clickjacking?

XSS: уникати dangerouslySetInnerHTML, використовувати Content Security Policy (CSP), escape дані перед рендером.

CSRF: застосовувати SameSite cookies, токени (CSRF token), перевірку реферера.

Clickjacking: заборонити вбудовування сторінки через заголовок X-Frame-Options: DENY або frame-ancestors у CSP.

32. Як працює Content Security Policy (CSP) і як її правильно налаштувати?

CSP — політика, що обмежує джерела завантаження ресурсів, щоб запобігти XSS.

Вказується у заголовку:

Content-Security-Policy: default-src 'self'; img-src 'self' https:; script-src 'self' https://apis.google.com

Також можна додати report-uri або report-to для моніторингу порушень.

33. Як зберігати JWT або токени без ризику витоку?

Небезпечно: LocalStorage (можливість XSS-доступу).

Безпечніше: HttpOnly secure cookie + SameSite.

Альтернатива — зберігати у пам’яті програми (in-memory) і рефрешити при перезавантаженні через secure refresh-token cookie.

Set-Cookie: token=abc; HttpOnly; Secure; SameSite=Strict
34. Як перевірити, що API виклики не піддаються маніпуляціям з боку клієнта?

На клієнті — ніяк повністю не гарантуєш безпеку. Перевірки повинні бути на сервері:

  • JWT підпис (HMAC або RSA).
  • Перевірка ролей та доступів (RBAC/ABAC).
  • Rate limiting, CORS, request validation.

Для додаткової безпеки — серверна валідація payload і timestamp-захист.

35. Як виявляти витоки даних у фронтенд-логах або аналітиці?

Не логувати персональні або секретні дані (PII, токени, паролі).

Використовувати Data Masking і Sentry/Datadog з фільтрацією sensitive інформації.

Sentry.init({
  beforeSend(event) {
    delete event.request?.cookies;
    return event;
  }
});
36. Як працює SameSite cookie і чим відрізняється Strict / Lax / None?

Strict: cookie надсилається тільки з того ж домену (найбезпечніше).

Lax: cookie надсилається при навігації користувача (за посиланням).

None: cookie доступне скрізь, але потрібно додати Secure.

Set-Cookie: session=123; SameSite=Lax; Secure; HttpOnly
37. Які основні загрози безпеці у SPA ти бачив на практиці?
  • XSS через неконтрольований HTML або сторонні віджети.
  • CSRF при використанні LocalStorage токенів.
  • Несанкціоновані запити до API без перевірки ролей.
  • Витік даних через аналітику, logs або query params.
  • Hardcoded API keys у коді фронтенду.
38. Як ти підходиш до структуризації великого React-проєкту (modules, features, slices)?

Застосовую feature-based architecture або atomic design:

  • src/features — бізнес-фічі (auth, users, posts)
  • src/entities — сутності з моделями даних
  • src/shared — утиліти, UI, hooks
  • src/pages — маршрути

Кожна фіча має свої slice/store/tests/стилі — це спрощує масштабування.

39. Як би ти реалізував feature flags у фронтенді без деплою?

Використовую remote config (наприклад, Firebase Remote Config або власне API).

Фічі активуються через бекенд або JSON-конфіг, що кешується локально. Можна керувати через LaunchDarkly або ConfigCat.

const flags = await fetch('/flags.json').then(r => r.json());
if (flags.newUI) render(<NewLayout />);
40. Як ти організовуєш версіонування API і backward compatibility у фронті?

Використовую версіювання у URL або через заголовки (Accept: application/vnd.api.v2+json).

Для backward compatibility — feature flags, адаптери для старих endpointів, поступове видалення deprecated API.

GET https://api.example.com/v2/users
41. Як проєктувати Error handling у масштабному застосунку?

Підхід “global + local handling”:

  • Глобальні Error Boundaries для React-UI.
  • Глобальний перехоплювач API-помилок (axios interceptors, fetch wrapper).
  • Сповіщення користувачу через Toast або modal.
  • Логування через Sentry, LogRocket, Datadog.
axios.interceptors.response.use(null, (error) => {
  notify(error.message);
  return Promise.reject(error);
});
42. Як зробити мікрофронтенди — і коли це реально виправдано?

Мікрофронтенди — розподіл великого SPA на незалежні модулі, які можуть розроблятись і деплоїтись окремо.

Реалізація: Module Federation (Webpack 5), single-spa, iframe/iframe-less shell.

Виправдано: великі команди, ізольовані фічі, часті деплої. Але збільшує складність DevOps та комунікації між модулями.

// host webpack.config.js
new ModuleFederationPlugin({
  remotes: { profile: "profileApp@http://localhost:3001/remoteEntry.js" }
});
43. Як ти відстежуєш перформанс у продакшені (Sentry, Datadog, Perfume.js)?

Використовую Real User Monitoring (RUM) інструменти:

  • Sentry Performance — trace API-викликів, рендери, React commit time.
  • Datadog / NewRelic — розподілений трейсинг, метрики ресурсів.
  • Perfume.js або Web-Vitals — LCP, FID, CLS у runtime.
import Perfume from 'perfume.js';
new Perfume({ analyticsTracker: ({ metricName, data }) => {
  sendToAnalytics(metricName, data);
}});

46. Які типові анти-патерни у React-коді ти бачив і як їх уникнути?
  • Зберігання несеріалізованих об’єктів у state → некоректні порівняння.
  • Виклик setState у кожному рендері → нескінченний цикл.
  • Залежності useEffect без масиву або з зайвими полями.
  • Відсутність keys у списках.
  • Великий контекст, що тригерить усі ререндери.

Рішення: мемоізація, селектори, DevTools Profiler, ESLint-rules (“react-hooks/exhaustive-deps”).

47. Як ти приймаєш рішення між оптимізацією і читабельністю коду?

Залежно від контексту:

  • Критичні ділянки (scroll, rendering, loops) — пріоритет перформансу.
  • Бізнес-логіка / UI — пріоритет підтримуваності.

Завжди починаю з читабельного рішення → профілюю → оптимізую лише вузькі місця. Пишу коментарі, якщо оптимізація неочевидна.

Різниця між SSR, ISR та CSR в NextJS Це три різні способи коли і де формується HTML сторінки. Або простіше: коли саме тобі “готують” UI. 🍽️ CSR (Client-Side Rendering) Аналогія: Тобі дають порожню тарілку й інгредієнти - все готується прямо на столі в браузері. Технічно: • Сервер віддає порожній index.html + JS бандл • React монтується на клієнті • Дані фетчаться після завантаження JS • Перший meaningful paint - довший ✔ мінімальне навантаження на сервер ✖ гірший SEO ✖ повільний перший рендер 👉 Класична SPA (Create React App, Vite + React). 🔥 SSR (Server-Side Rendering) Аналогія: Шеф готує страву на кухні. Все готово ще до того, як ти сів за стіл. Технічно: • Сервер рендерить HTML на кожен запит • React виконується на сервері → віддає вже готову HTML-структуру • Після приходу на клієнт - гідратація (React “оживляє” HTML) • Дані завжди актуальні, бо фетчаться на сервері on request ✔ ідеально для SEO ✔ швидкий перший піксель ✖ навантаження на сервер росте з трафіком ✖ гідратація може бути повільною 👉 Next.js: getServerSideProps, server components, Route Handlers. ⚡ ISR (Incremental Static Regeneration) Аналогія: Приготували 100 порцій наперед. Якщо порція “прострочена” - готують нову. Ти все одно отримуєш швидко. Технічно: • Сторінка генерується один раз під час білду або першого запиту • Далі кешується в CDN як статичний HTML • Через revalidate: X Next.js фоново генерує нову версію • Трафік йде по CDN → ультра швидко ✔ швидкість статичного сайту + динамічність ✔ менше навантаження на бек ✔ нормальний SEO ✖ не підходить для highly dynamic контенту (live data) 👉 Next.js: fetch(..., { next: { revalidate: 60 } }), generateStaticParams. 🔥 Пост з прикладами та відео: https://lnkd.in/dnHC5t4t
